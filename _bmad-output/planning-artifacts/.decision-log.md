# Decision Log — VacationRequest
*Updated 2026-06-11*

## Brief Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Supervisor vacations are out of scope | Supervisors are committee members, not employees under the same contract structure |
| 2 | Fallback approvers are configurable | Prevents requests from being blocked when a supervisor is unavailable |
| 3 | Entitlement is set manually per year by each supervisor | Calculation rules (sex, age, seniority) are too variable to automate reliably in v1 |
| 4 | Unused days roll over and are explicitly labelled | Contractual requirement; carry-over must be distinguishable from fresh entitlement |
| 5 | Rejection requires a mandatory reason | Provides audit trail and fairness for trainers |
| 6 | Supervisors can revoke approved requests | Operational reality — plans change |
| 7 | All supervisors have admin rights | Club is small; no need for a separate IT admin role |
| 8 | English only | Club operating language chosen; no multi-language requirement raised |
| 9 | Web + mobile responsive (no native apps) | Covers all use cases without native app development cost |
| 10 | All lifecycle events trigger both in-app and email notifications | No events were flagged as notification-exempt |

## PRD Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| 11 | Trainer and Supervisor are mutually exclusive roles | Confirmed explicitly; no dual-role scenario exists |
| 12 | One supervisor per trainer; supervisor can have multiple trainers | Club structure; simplifies approval routing |
| 13 | Fallback approver can only approve/reject pending requests | Full admin delegation is unnecessary and risky for a deputy role |
| 14 | Draft state exists before submission | Allows trainers to prepare requests without triggering workflow |
| 15 | No restriction on resubmitting after rejection or revocation | No business rule preventing it; trainer's right to retry |
| 16 | System blocks overlapping requests | Prevents double-booking; enforced at submission time |
| 17 | Calendar year (Jan 1 – Dec 31) for entitlement | Standard Swiss employment practice |
| 18 | 10-day grace period for entitlement edits, then locked | Allows corrections after initial entry; locks for compliance |
| 19 | Carry-over expires March 1 (end of 2-month grace window) | Gives trainers time to use carry-over; club's stated policy |
| 20 | Cross-year requests deduct from the year the days fall in | Logical and fair; avoids gaming year boundaries |
| 21 | Notification: actor gets confirmation, other party gets notification | Clear, predictable model; no silent actions |
| 22 | Pending reminder fires after 5 working days | Supervisor stated threshold; balances urgency with admin load |
| 23 | Carry-over reminder fires February 1 | One month before expiry gives trainers adequate time to act |
| 24 | Trainer sees own balance only | Privacy; no need to compare with colleagues |
| 25 | Supervisor sees all assigned trainers' balances | Required for entitlement management and approval decisions |
| 26 | Email/password auth, admin creates accounts, no MFA (v1) | Simplest viable auth; explicitly flagged for change request |
| 27 | Data retention 5 years | Swiss employment law compliance |
| 28 | Local hosting for v1 | Simplest deployment; explicitly flagged for change request |
| 29 | Internal audit log visible to supervisors | Compliance requirement; read-only |
